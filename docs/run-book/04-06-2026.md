## Detailed Discussion: Why Extraction Initially Failed

The first extraction attempt failed for two distinct operational reasons encountered in sequence.

## Failure A: Remote removal assumption mismatch after `filter-repo`

What happened:
- The scripted sequence included `git remote remove origin`.
- In the disposable clone, `git filter-repo` had already removed/rewritten remote metadata state such that `origin` no longer existed.
- Command produced: `error: No such remote: 'origin'`.

Why this occurred:
- `git filter-repo` intentionally sanitizes repository metadata/history context during rewrite operations.
- In disposable clones this is common and safe, but scripts that unconditionally remove `origin` can fail if `origin` is already absent.

How it was handled:
- Continued safely by directly adding `origin` to target repo and proceeding.

Key lesson:
- Post-filter scripts should be idempotent around remote setup (e.g., tolerate absent `origin`).

## Failure B (primary root cause): incorrect subdirectory filter path

What happened:
- Initial command used:
  `git filter-repo --subdirectory-filter mempool-ipc/ --force`
- In source repo, the canonical data-plane path was actually:
  `software/mempool-ipc/`
- The filtered result therefore did not produce the expected extracted history/tree.

Why this occurred:
- Path assumption from historical repo layout did not match final monorepo layout.
- Subdirectory filtering is exact-path based; wrong prefix rewrites to wrong/empty outcome.

Symptoms observed:
- Successful-looking push but resulting branch had only bootstrap history/files.
- No expected data-plane directories in final tree.

How it was resolved:
1. Audited source tree paths across branches.
2. Confirmed true path `software/mempool-ipc/`.
3. Re-ran extraction from a fresh clone with corrected filter path.
4. Re-merged and force-pushed corrected branch.

Key lesson:
- Always verify source path via tree query before running irreversible history rewrite.

## Failure C: README merge conflict after unrelated-history merge

What happened:
- Corrected extraction plus bootstrap branch merge caused `README.md` add/add conflict.

Why this occurred:
- Both histories introduced distinct top-level README content.

How it was resolved:
- Produced a merged README combining standalone m-ipc context with recovered technical detail.

Key lesson:
- For decoupling migrations, README conflicts are expected and should be resolved intentionally, not via one-sided overwrite.

## Recommended Guardrails for Future Extractions

1. Validate source path before rewrite:
   - `git ls-tree -r --name-only <ref> | grep <expected-subdir>`
2. Use disposable fresh clone for every rewrite.
3. Treat post-filter remote operations as idempotent.
4. Verify branch contents before push:
   - directory list
   - commit count/log graph
5. After merge, explicitly review README/docs conflicts.
6. Verify remote refs after push (`git ls-remote --heads ...`).
